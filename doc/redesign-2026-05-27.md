# kenos 再設計 (2026-05-27)

メタ会話で固まった、task skill とその周辺の再設計。実装は別session。

## 背景: 3つのテーマ

1. **情報の分割** — log.md が肥大化しすぎて、AIが毎回読むコストが高い + 人間が読みづらい
2. **コスト削減** — session再開時に log.md 全体を読まされるのが重い
3. **作業分担** — 表(実作業)と裏(振り返り)を分離して、フォア Claude を実作業に集中させたい

3つは別物に見えるが、実際は **「log.md が consumer 別の情報を全部抱えている」** という同じ根に行き着く。分割で全部解ける。

---

## A. 新しいファイル構造

```
.tasks/<ticket>/
├── overview.md     # 設計図 (背景・ゴール・スコープ・問題・課題 集約)
├── roadmap.md      # 順序つきTODO (チェックリスト形式)
├── issues.md       # 論点・人に聞くこと (見出し+議論形式)
├── followup.md     # スコープ外で見つけたもの
├── log.md          # 時系列メモ (裏 Claude が更新)
├── habits.md       # 癖の観察 (裏 Claude が記録)
└── knowledge/      # 調査・設計知識 (ad hoc命名、ticket内閉じ)
```

### 各ファイルの役割と更新の仕方

| ファイル | 更新方法 | 主な読み手 |
|---|---|---|
| `overview.md` | 漸近的に更新(現状把握が進むたび) | session 再開時のAI/人間、振り返り |
| `roadmap.md` | 順序が動的に変わる、チェック消化 | session毎に表 Claude が更新 |
| `issues.md` | 論点が出たら追加、解消したら消す | 表 Claude / 人間 |
| `followup.md` | 見つけたら追加、ticket完了後に引き渡し | ticket後の人間 |
| `log.md` | append-only | 振り返り時のみ |
| `habits.md` | append-only | 振り返り時のみ |
| `knowledge/*` | トピック単位で追加 | 調査結果を再利用する時 |

### 廃止するもの

- `session.md` — overview + roadmap + issues の3点で session再開に必要な情報が揃うため不要
- `progress.md` (旧名) — `roadmap.md` に改称
- `questions.md` (旧名) — `issues.md` に改称(論点も含める意味で広く)

### 判断軸: なぜこの分割か

**更新頻度 × 読み手 × 構造(リスト型 vs 議論型)** で切ると自然に分かれる:

- **不変系** (`overview`): タスクの設計図。漸近的にだが大きくは変わらない
- **状態系** (`roadmap`, `issues`, `followup`): 現在の状態を保つ。表 Claude が随時更新
- **記録系** (`log`, `habits`): append-only。振り返り用。裏 Claude が更新
- **知識系** (`knowledge/`): 調査の成果物。ad hoc

**roadmap と issues を分けた理由**: 同じ "やること" でも構造が違う。roadmap は順序つきチェックリスト、issues は見出しつき議論。1ファイルに混ぜると片方の形に引っ張られて表現力が落ちる。

**knowledge を ticket 内に閉じた理由**: 横断的整理は振り返りで後からやればよい。最初から横断構造を作ると振り分け判断が増えて面倒(YAGNI)。

## B. /task skill の動作変更

### 新規フロー

1. URLから ticket番号を抽出
2. `.tasks/<ticket-no>/` 作成
3. ticket本文取得(MCP)
4. 上記7ファイル/dir を雛形で生成
5. `.kenos/current-ticket` に `.tasks/<ticket-no>/` のパスを書く ← **新規(裏Claude機構用)**
6. 現在地把握フェーズへ(`overview.md` を埋める)

### 再開フロー

1. `.tasks/<ticket-no>/overview.md`, `roadmap.md`, `issues.md` の3つを読む(log.md は読まない)
2. 現在地・次にやること・未解決論点を表示
3. `.kenos/current-ticket` を更新
4. 「ここから再開します。何から手をつけますか?」

**コスト削減効果**: 旧 log.md(数百〜千行)を毎回読む → 3ファイル合計で数十〜百行程度。AIの読み込みコストが大幅に下がる。

## C. 癖の指摘ロジック更新

### 旧ルール

> 計画やTODOリストを立てた直後、**必ず**以下の問いを出す。形骸化してもいいので、毎回聞く。

### 新ルール

**節目・迷い・分岐で発火、確定言葉が出てる時は聞かない**。

#### 発火する場面

- タスク開始時(現在地把握 〜 計画確定まで)— 必ず聞く
- 方針転換・大きな分岐(やり方の選択肢が複数ある時)
- Ken が迷い言葉を出した瞬間 (「どっちにしようかな」「悩んでる」「わからない」)
- 完了直前の最終確認

#### 発火しない場面

- 実行フェーズで Ken が確定言葉で話してる時 (「〜でいく」「〜やる」「決めた」)
- ルーチン的な進捗確認・手を動かしてるフェーズ

### 判断軸

「形骸化しても毎回」は **解決フェーズでも飛んでくる** ので邪魔になる。「迷ってる瞬間に問う」に反転させる。リアルタイム矯正の価値は残しつつ、無駄な介入を削る。

## D. 裏Claude 機構(新規)

表/裏の分担:
- **表 Claude**: overview / roadmap / issues 更新 + 実作業 + 癖の指摘(節目で)
- **裏 Claude**: log.md (時系列メモ) と habits.md (癖の記録) を更新

### 機構: Claude Code hooks

`.claude/settings.json`:

```json
{
  "hooks": {
    "SessionEnd": [{
      "matcher": "",
      "hooks": [{ "type": "command", "command": ".claude/hooks/session-end.sh" }]
    }],
    "SessionStart": [{
      "matcher": "",
      "hooks": [{ "type": "command", "command": ".claude/hooks/session-start.sh" }]
    }]
  }
}
```

### `.claude/hooks/session-end.sh`

```bash
#!/bin/bash
input=$(cat)
transcript=$(echo "$input" | jq -r '.transcript_path')
ticket_dir=$(cat .kenos/current-ticket 2>/dev/null) || exit 0

nohup claude -p "
Read transcript at: $transcript
Append brief timeline entry to $ticket_dir/log.md
If Ken's habits appear (good or bad), record to $ticket_dir/habits.md
Stay concise.
" > /tmp/kenos-bg.log 2>&1 &
disown
echo "$transcript" > .kenos/last-processed
exit 0
```

**ポイント:**
- `nohup ... &` + `disown` — 表 Claude が裏の完了を待たない(これがないと作業が止まる)
- `claude -p` — Claude Code の non-interactive モード
- 状態ファイル `.kenos/current-ticket` で書き込み先を伝える

### `.claude/hooks/session-start.sh` (取りこぼし回収)

PC強制終了などで SessionEnd が走らなかったケース対策。

```bash
#!/bin/bash
last=$(cat .kenos/last-processed 2>/dev/null || echo "")
transcripts=~/.claude/projects/<encoded-cwd>/

unprocessed=$(ls -t $transcripts/*.jsonl | awk -v last="$last" '$0 > last')
if [ -n "$unprocessed" ]; then
  latest=$(echo "$unprocessed" | head -1)
  nohup claude -p "Read $latest. Append to log/habits as if SessionEnd had fired." > /tmp/kenos-bg.log 2>&1 &
  disown
  echo "$latest" > .kenos/last-processed
fi
exit 0
```

複数取りこぼしがあった場合は **最新1つだけ自動拾い**。古いものは `kenos reflect` で手動回収。

### Stop hook を使わなかった理由

`Stop` (毎応答ごと) は細かく log.md が更新できる代わりに、毎応答で裏 Claude が起動 → コスト過大。`SessionEnd` で1session 1回にまとめる方が実用的。

## E. kenos CLI 追加コマンド

- `kenos reflect [transcript-path]` — 手動振り返り発火(取りこぼし救済 / 古い transcript の後追い)
- (optional) `kenos bg log` — 裏 Claude のログ(`/tmp/kenos-bg.log`)閲覧

## F. payload 拡張

`payload/skills/` に加えて以下を追加:

```
payload/
├── skills/
│   ├── task/SKILL.md      ← 更新
│   └── task-sync/SKILL.md  ← 必要なら更新
├── hooks/
│   ├── session-end.sh
│   └── session-start.sh
└── settings.json          ← hooks 登録テンプレート
```

`kenos init` で hooks も配線する。既存の `.claude/settings.json` がある場合はマージか確認が必要。

## 実装の進め方(別sessionで)

優先順:

1. **ファイル構造の変更** (`payload/skills/task/SKILL.md` 更新) — まずここ
2. **/task skill の動作変更** (新規フロー / 再開フロー)
3. **癖の指摘ロジック更新** (SKILL.md内の該当節を書き換え)
4. **裏Claude機構** (hooks スクリプト + payload追加 + `kenos init` で配線)
5. **kenos CLI**: `kenos reflect` 追加
6. **README.md 更新**

1-3 は SKILL.md だけの変更で動作確認できる。4-5 は CLI と payload まで広がるので段階を分ける。

### 後方互換について

既存の .tasks/ 配下にある旧形式の log.md(全部入り)は **そのまま残す**。新規 ticket から新形式で運用。移行スクリプトは作らない(必要になってから考える)。

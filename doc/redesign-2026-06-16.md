# kenos 再設計 (2026-06-16)

メタ会話で固まった、`.tasks/<ticket>/` 配下のファイル分担の再設計。実装は別session。

## 背景

現状の `roadmap.md` と `issues.md` の運用で起きてること:

- **roadmap が細かい TODO に侵食される** — roadmap は本来「方向」を描く場所だが、順序つき TODO として運用してると、調査が進むごとに細かいタスクが書き足されて肥大化する。長期的なほど不確実なのに、細かいタスクを遠くまで書くのは無理がある。
- **issues と TODO が混同される** — 「やること」っぽいものは何でも roadmap か issues に入ってしまう。本来 issues は「問い」を入れる場所で、TODO とは違う性質。
- **入力負荷** — issues を見出し+議論形式にすると、書き出すハードルが高い。気になったことをサッと書ける形になっていない。

3つを分けて役割を固定すると解ける。

---

## A. 新しいファイル構造

```
.tasks/<ticket>/
├── overview.md     # 設計図 (変更なし)
├── roadmap.md      # 方向性・大きな道  ← 役割変更
├── now.md          # 今のフェーズの細かい TODO  ← 新規
├── issues.md       # 問いの inbox (flat bullet)  ← 役割変更
├── followup.md     # スコープ外で見つけたもの (変更なし)
├── log.md          # 時系列メモ (変更なし)
├── habits.md       # 癖の観察 (変更なし)
└── findings/       # 調査結果 (research/ から改称)  ← リネーム
```

### 各ファイルの役割

| ファイル | 役割 | 形式 |
|---|---|---|
| `roadmap.md` | 方向性。フェーズの大きな区切り。長期になるほど抽象的でよい | フェーズの並び |
| `now.md` | 今のフェーズで実際に動くタスク。細かい TODO はここ | チェックリスト |
| `issues.md` | 問い・リスク・気になってること・疑問の inbox。入力負荷を下げる | flat bullet |
| `findings/` | 調査・調べた結果。自然な単位で分割、issue に紐付けない | ad hoc 命名 |

### 判断軸: なぜこの分け方か

**「方向 / 行動 / 問い / 知見」の4軸で切る**:

- **方向** (`roadmap`): 不確実な未来を抽象的に。細かく書こうとしない
- **行動** (`now`): 今手を動かすために必要な粒度。完了したら消える
- **問い** (`issues`): 答えるべきこと、気になってること。bullet で書き出すだけ
- **知見** (`findings/`): 調べてわかったこと。再利用される可能性のあるもの

### issues → now の関係

issues は「移送」しない。

- そのフェーズで解消すべき問いがあれば、now にタスクが現れる(手で書く or 思いついた時)
- now のタスクが完了すると、関連する issue が(頭の中で)解消される
- 解消された issue は issues.md から消す

ファイル上で issue と now のタスクを紐付けない(Kenの頭の中で繋ぐ)。番号付けは見送り(必要になったら追加)。

### findings の粒度

- issue に縛られない自然な単位で分割
- 1 finding が複数 issue にまたがってよい
- ad hoc な命名(`research/` と同じ運用)

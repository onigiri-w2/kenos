# 仕組みアイデアの棚

このプロジェクトの会話中に思いついた「仕組み案」「改善案」を保管する場所。
**ここに置くこと自体は歓迎**。今やるかどうかは別の判断軸。
棚を見返すタイミングは、tickets が空いたメタ時間、または明らかに関連する話題が出た時。

## 雛形

```markdown
## <アイデアのタイトル>

- **思いついた場面**: どういう会話/観察から出てきたか
- **内容**: 何をする案か(1-3行)
- **すぐできるか**: Yes/No、その理由
- **今自分に時間があるか**: (棚を見返す時点で記入)
- **解消した時の影響**: これをやると何が変わるか。書けないなら「まだ不明」でOK
- **状態**: 棚に保管中 / 着手中 / 完了 / 却下
```

## ticket一覧から選んで再開する仕組み + /task の新規/再開分岐

- **思いついた場面**: 2026-04-24 メタ会話中、「途中で再開するタスクを毎回 `/task` から起動するのがしんどい。ticket ID を Doc から拾うのも面倒」という実感から。深掘りしていくと、再開が遅い原因は `/task` が新規前提で動いてる(ディレクトリ作成、Jira取得、雛形生成)からだと判明した
- **内容**: 2つ組み合わせる案。
  1. 外側のshell(fzf等)で `.tasks/` 配下を一覧表示 → 選んで `claude` 起動 → `/task <ID>` 実行
  2. `/task` の最初で `.tasks/<ticket-no>/` の有無をチェック。あれば log.md を読むだけの軽い再開フロー、なければ今の新規フロー
- **すぐできるか**: Yes。実装タスクなので Claude Code 側で動かしながら作る方が早い
- **今自分に時間があるか**: (判断時に記入)
- **解消した時の影響**: 再開時の「ID探す手間」と「新規処理が走る遅さ」の両方が消える。新規ticketの時は従来通り `/task <URL>` を手で叩く
- **状態**: 棚に保管中(実装タスクなので Claude Code 側でやる)

## ken-workflow を Claude Code plugin 化する

- **思いついた場面**: 2026-04-24 メタ会話中、複数pjtで同じskillを使い回したい話の延長で。最初はsymlinkで本体を一箇所に置く案だったが、「新pjtができた時にスッと差し込める」思想ならpluginの方が合うという話になった
- **内容**: `~/ken-workflow/` を Claude Code の plugin(かつローカル marketplace)として構成する。新pjtでは `/plugin marketplace add ~/ken-workflow` → `/plugin install workflow` で差し込む。marketplace公開は不要、ローカルパスでOK
- **すぐできるか**: No。marketplace.json / plugin.json の構造を調べて書く必要がある。skillが増えて symlink運用が面倒になってから着手で十分
- **今自分に時間があるか**: (判断時に記入)
- **解消した時の影響**: 新pjtに手動で `ln -s` を張る作業がなくなる。skillの追加・更新も marketplace 側で一元化される。将来他の人と共有したくなった時もそのままGitHub pushするだけで切り替えられる
- **状態**: 棚に保管中(今日は `~/ken-workflow/skills/` に実体を引っ越すところまで。plugin化はskillが増えてから)

## 運用メモ

- **追加する時**: Claude が会話中に artifact として更新する → Ken が Knowledge にアップロードして上書きする
- **見返す時**: ticket が空いたメタ時間に、このプロジェクトを開いて一緒に見る
- **却下する時**: 項目を消さずに `状態: 却下` に変える。理由も書く(後で「なぜ却下したか」を拾えるように)

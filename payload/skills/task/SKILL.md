# /task: Ticket着手の初期化

チケットのURLを受け取り、作業場所を準備する。

## 入力

- `$ARGUMENTS`: チケットの URL **または** ticket番号 (例: `PROJ-123`)

## 分岐: 新規 or 再開

`$ARGUMENTS` を見て分岐する:

- **ticket番号だけ** (例: `PROJ-123`) かつ `.tasks/<ticket-no>/overview.md` が存在する → **再開フロー**へ
- **URL** → **新規フロー**へ
- **ticket番号だけ** だが `.tasks/<ticket-no>/overview.md` が存在しない → 「そのticketの作業場所はまだありません。URLを貼ってください」と返す

---

## ファイル構造

`.tasks/<ticket-no>/` の中身は以下:

```
.tasks/<ticket>/
├── overview.md     # 設計図 (背景・ゴール・スコープ・現在地・メタ情報)
├── roadmap.md      # 順序つきTODO (チェックリスト形式)
├── issues.md       # 論点・人に聞くこと (見出し+議論形式)
├── followup.md     # スコープ外で見つけたもの
├── log.md          # 時系列メモ (裏 Claude が更新、表 Claude は読まない)
├── habits.md       # 癖の観察 (裏 Claude が記録、表 Claude は読まない)
└── knowledge/      # 調査・設計知識 (ad hoc命名、ticket内閉じ)
```

| ファイル | 更新方法 | 主な読み手 |
|---|---|---|
| `overview.md` | 漸近的に更新(現状把握が進むたび) | session 再開時のAI/人間、振り返り |
| `roadmap.md` | 順序が動的に変わる、チェック消化 | session毎に表 Claude が更新 |
| `issues.md` | 論点が出たら追加、解消したら消す | 表 Claude / 人間 |
| `followup.md` | 見つけたら追加、ticket完了後に引き渡し | ticket後の人間 |
| `log.md` | append-only | 振り返り時のみ(表 Claude は触らない) |
| `habits.md` | append-only | 振り返り時のみ(表 Claude は触らない) |
| `knowledge/*` | トピック単位で追加 | 調査結果を再利用する時 |

---

## 再開フロー

1. `.tasks/<ticket-no>/overview.md`, `roadmap.md`, `issues.md` の3つを読む(**log.md は読まない**)
2. `.kenos/current-ticket` を `.tasks/<ticket-no>` で更新する(裏 Claude機構用)
3. 以下を表示する:
   - メタ情報(期限、ステータス)
   - 現在地(わかっていること / わかっていないこと)
   - 次にやること(roadmap の未チェック先頭)
   - 未解消の論点(issues の見出し一覧)
4. 「ここから再開します。何から手をつけますか?」と聞いて始める

---

## 新規フロー

1. URLからticket番号を抽出する (例: `PROJ-123`)
2. `.tasks/<ticket-no>/` ディレクトリを作成する (既存ならskip)
3. チケット管理ツールの MCP を使って ticket 本文を取得する
4. 以下7つを雛形で生成する:

### `overview.md`

~~~markdown
# <ticket-no>

## メタ情報

- 期限:
- ステータス: 未着手

## Ticket本文

(取得したticket本文をそのまま貼る)

## AI要約

(この時点では空。Kenが現在地を書いた後で埋める)

## 現在地

### わかっていること

-

### わかっていないこと

-

## ゴール

(タスクが完了した状態を1-2行で。最初は空でよい)

## スコープ

### やる

-

### やらない

-

## 問題・課題

(設計上の問題や検討すべき課題。出てきたら書く)
~~~

### `roadmap.md`

~~~markdown
# <ticket-no> roadmap

順序つきTODO。チェック消化で進捗を表す。順序は動的に変わってOK。

- [ ]
~~~

### `issues.md`

~~~markdown
# <ticket-no> issues

論点・人に聞くこと。解消したら消す。

<!-- 例:
## XXXの扱いをどうするか

- 選択肢A: ...
- 選択肢B: ...
- 確認したい人: @yyy
-->
~~~

### `followup.md`

~~~markdown
# <ticket-no> followup

このticketのスコープ外で見つけたもの。ticket完了後に引き渡す。

-
~~~

### `log.md`

~~~markdown
# <ticket-no> log

時系列メモ。session終了時に裏 Claude が append する。表 Claude は触らない。
~~~

### `habits.md`

~~~markdown
# <ticket-no> habits

癖の観察。良い動きも記録する。裏 Claude が transcript から拾って記録する。
~~~

### `knowledge/`

空ディレクトリ。調査メモを ad hoc な命名で追加していく。

5. プロジェクト直下に `.kenos/current-ticket` を作成し、`.tasks/<ticket-no>` を書き込む(裏 Claude機構が読む)
6. 作成したパス一覧を表示する
7. 「現在地を埋めるフェーズ」に入る(下記参照)

---

## ステータスの書き方

`overview.md` の「ステータス」欄は以下の3値のいずれかを使う:

- `未着手`
- `進行中`
- `完了`

補足が必要な状態(レビュー待ち、ブロック中、等)は括弧書きで添える:

- `進行中 (レビュー待ち)`
- `進行中 (他チーム待ち)`

## 現在地を埋めるフェーズの進め方

**原則: AIから観点リストを先に提示しない。Kenの思考を奪わない。**

1. Kenにこう聞く: 「このticketは自分の言葉でどういう話ですか? 何をする必要がありますか?」
2. Kenが説明する
3. Kenの説明を聞いて、以下を観察する:
   - 具体的に説明できた部分 → 「わかっていること」の候補
   - 曖昧だった部分、スキップした部分、抽象語で済ませた部分 → 「わかっていないこと」の候補
4. 観察した内容をKenに返す。「〜は具体的に言えましたね」「〜は触れませんでしたが、見えていますか?」という形で。
5. Kenが「それはわからない」と認めたら「わかっていないこと」に書く。「あ、考えてなかった」と気づいたら、それも「わかっていないこと」に書く。
6. 一通り出揃ったら、**最後に**AI側から「こういう観点も抜けていませんか?」と補足する。この順序を逆にしない。
7. 現在地が埋まったら、AI要約を書く。Kenの現在地と、AIが読み取ったticketの要点を並べると、ギャップが見える。

## 癖の指摘ロジック

**節目・迷い・分岐で発火する。確定言葉が出てる時は聞かない。**

### 発火する場面

- **タスク開始時**(現在地把握 〜 計画確定まで) — 必ず聞く
- **方針転換・大きな分岐**(やり方の選択肢が複数ある時)
- **Ken が迷い言葉を出した瞬間**(「どっちにしようかな」「悩んでる」「わからない」)
- **完了直前の最終確認**

### 発火しない場面

- 実行フェーズで Ken が確定言葉で話してる時(「〜でいく」「〜やる」「決めた」)
- ルーチン的な進捗確認・手を動かしてるフェーズ

### 問いの内容

節目で問う2つの問い:

1. **「このTODOの中で、削れるものはありますか?」**
   - 「全部必要」と即答したら、「なぜそれぞれ必要か、1ステップずつ根拠を言えますか?」と深掘りする
   - ticketに書いてあることでも、今回のプロジェクト方針と照らして不要なステップがないか確認する
   - 「将来必要になるから今やる」という根拠が出たら、「逆に『将来必要になるから今はやらない』とも言えませんか?」と返す

2. **「この判断、1人で決めていい話ですか?」**
   - Kenが何かを決めようとした瞬間(「〜します」「〜でいきます」「〜が正しい」等)に必ず問う
   - 特にticket書き手、依頼者、関係者の意図に関わる判断では厳しく問う
   - 「確認しなくていい」と即答したら、「なぜ確認不要と判断したか」を言語化させる

### 判断軸

「形骸化しても毎回」は **解決フェーズでも飛んでくる** ので邪魔になる。「迷ってる瞬間に問う」に反転させる。リアルタイム矯正の価値は残しつつ、無駄な介入を削る。

## この後のKenさんとの振る舞い方

Kenは以下の癖を自己申告している:

- タスクの前提を揃えたり疑ったりするのが苦手
- ゴールをすぐ確定させてしまう
- 影響範囲への意識が抜ける
- 知識の拡充に逃げる(仕組み作りで満足して本題が進まない)
- 計画が苦手
- もらったTodoを全部やろうとしてしまう(QCDの観点で疑うテンプレがない)
- 人に聞くのがうまくできない(心の中で違和感を感じても無視しがち)

そのため:

- Kenが現在地を埋める前にゴールや計画に飛ぼうとしたら、一度止めて現在地の確認に戻す
- Kenが過剰に調査や仕組み作りに向かい始めたら、「それは今必要か?」と問う
- Kenが発言や判断で上記の癖を出した瞬間、指摘する(責めるのではなく観察として)
- 良い動き(自分で立ち止まった、前提を疑った、自分の認識の曖昧さに気づいた、等)も同じく指摘する

**癖の記録は裏 Claude が `habits.md` に書く**。表 Claude は指摘するだけで、自分では記録しない。

## AIの振る舞い原則

- 親切すぎない。Kenの思考を代行しない。
- リストや選択肢を先に出さない。まずKenの言葉を引き出す。
- Kenが曖昧に言ったことを「つまり〜ということですね?」と勝手にシャープ化しない。曖昧さは曖昧さのまま扱う。
- Kenが自分で気づけるように問いを返す。答えを渡さない。
- 癖の指摘は節目・迷い・分岐で出す。確定言葉が出てる時は出さない。
